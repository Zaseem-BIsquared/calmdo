## Generator Interface & YAML Design

**User's key insight:** Rails/Phoenix CLI interface is not friendly — mistakes are hard to correct. User wants an interactive wizard that writes a YAML file, so the YAML becomes the editable source of truth.

**User's workflow vision:** Sitting with clients, iterating on data model, re-running generators repeatedly. YAML is a living specification.

| Option | Description | Selected |
|--------|-------------|----------|
| Co-located in src/features/{name}/ | YAML lives alongside generated code | ✓ |
| Root features/ directory | Separate top-level dir | |
| Inside templates/ | All generator stuff together | |

**User's choice:** Co-located with the feature.

| Option | Description | Selected |
|--------|-------------|----------|
| Smart merge/update | Try to preserve customizations on re-run | ✓ |
| Refuse and warn | Abort if files exist | |
| Overwrite with confirmation | Show diff, ask to confirm | |

**User's choice:** Smart merge — most ambitious but most valuable for iterative client workflow.

**Defaults discussion:** Team member concern — "we'll forget what defaults are." Explored:
- Verbose YAML with `# (default)` comments — rejected (too verbose)
- Shared `templates/defaults.yaml` reference file — accepted (clean, one place to look)
- Resolved YAML as committed artifact — accepted (full transparency)

**Final three-file chain:** defaults.yaml → {name}.gen.yaml → {name}.resolved.yaml (all committed)

## Relationship Handling

**User's key insight:** Need tight parent-child (ERP-style: PO header → PO items) AND loose association (task optionally linked to project). Research into Frappe's istable model confirmed the distinction.

| Concept | Frappe equivalent | Decision |
|---------|-------------------|----------|
| `children` (tight composition) | `istable: 1` (Table field) | ✓ Included |
| `has_many` (loose, required or optional) | Link field | ✓ Included |
| `belongs_to` | Reverse Link | ✓ Included |
| `embedded` | N/A (Convex-specific nested objects) | ✓ Included |

**Research conducted:** Convex parent-child patterns, Frappe istable/Table/Link model. Separate tables confirmed as default for Convex (no atomicity trade-off, better reactivity granularity).
