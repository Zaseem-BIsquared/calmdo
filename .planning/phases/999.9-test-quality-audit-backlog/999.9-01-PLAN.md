# Plan 999.9-01: Catalogue Pass — Audit Exclusions, v8 Ignores, and Test Quality

## Objective

Produce a comprehensive findings report by auditing every coverage exclusion in `vitest.config.ts`, every inline `v8 ignore` directive, and every test file against the testing philosophy. This gives full visibility into scope before committing to fixes.

## Why This Matters

D-09 mandates a two-pass approach: catalogue first, fix second. Blindly purging v8 ignores without understanding each one leads to thrashing — you remove an ignore, coverage drops, you discover the code is dead, you delete it, then realize another file depended on it. The catalogue pass prevents this by triaging everything upfront.

## must_haves
- [ ] Every `coverage.exclude` entry in `vitest.config.ts` triaged as `keep` (legitimate) or `remove` (needs tests) with reasoning
- [ ] Every `v8 ignore` block across all 30 files categorized by type (TanStack Form timing, Radix portal, dnd-kit, dead code, testable logic)
- [ ] Every test file (all 59) checked against the 12-point testing philosophy checklist
- [ ] Findings report written to `999.9-FINDINGS.md` with severity ratings (must-fix, should-fix, suggestion)
- [ ] Total estimated scope (number of files to change, tests to write) calculated for Plan 02 and 03

<title>Audit all v8 ignore directives across 30 files</title>
<type>analysis</type>

</read_first>
<action>
For each of the 76 `v8 ignore start` blocks, read the surrounding code and classify into one of these categories:

1. **TanStack Form re-render timing** (14 occurrences) — branches like `field.state.meta.errors.length > 0` that depend on TanStack Form's internal re-render cycle. Determine: can these be tested by triggering form validation via `userEvent.type` + `userEvent.click` and then asserting the error message appears? If yes, classify as `testable`. If the form validation truly cannot be triggered in jsdom, classify as `legitimate-ignore`.

2. **Radix portal/dropdown** (~15 occurrences) — code inside Radix Select, DropdownMenu, or Dialog portals. Determine per-component: can the component be rendered directly (outside the portal parent) and tested? Can `within()` target the portal content? If the component has logic worth testing, classify as `testable-via-direct-render`. If it's truly a pass-through wrapper, classify as `legitimate-ignore`.

3. **dnd-kit drag handlers** (4 occurrences) — drag event handlers that require pointer simulation. These are legitimately untestable in jsdom without a full browser. Classify as `legitimate-ignore` but note that E2E tests should cover drag-and-drop.

4. **Whole-file ignores** (4 files: TaskDetailPanel, SubtaskItem, SubtaskList, WorkLogForm, WorkLogList) — entire components wrapped in v8 ignore because they render inside a Sheet/Dialog portal. For each: does the component have testable logic (state management, conditional rendering, mutation calls)? If yes, classify as `testable-via-direct-render` — the component can be rendered directly in a test without the Sheet wrapper. If it's a pure layout component, classify as `legitimate-ignore`.

5. **Dead code / defensive guards** (~8 occurrences) — fallbacks like `|| ""` that can never be reached, defensive guards for disabled buttons, etc. Per D-05, classify as `dead-code-delete`.

6. **Server-dependent branches** (~5 occurrences) — `import.meta.env.DEV`, server error responses, signIn rejection paths. Determine: can the test mock the environment variable or simulate the error? If yes, classify as `testable`. If truly environment-locked, classify as `legitimate-ignore`.

7. **Spinner/loading states** (~5 occurrences) — spinner visibility during brief async operations. Classify as `testable` if `waitFor` can catch the loading state, or `legitimate-ignore` if the state is too transient.

<type>analysis</type>

</read_first>
<action>
Check every test file against the 12-point testing philosophy checklist. The checklist includes:

1. **Test behavior, not implementation** — assert outcomes/state changes, not mock calls
2. **Integration-first** — prefer real code paths over mocked ones
3. **MECE state decomposition** — mutually exclusive, collectively exhaustive test cases
4. **No v8 ignore as substitute for real tests** — (handled by Task 02, note here if test is missing because of v8 ignore)
5. **Each test independent and self-contained** — no test depends on another's state
6. **Proper naming** — `"verb what-user-sees [condition]"` format
7. **No snapshot tests** — assert specific values
8. **No `toBeDefined()`/`toBeTruthy()` as sole assertion** — assert specific values
9. **No `expect(true).toBe(true)` placeholders** — every test asserts real behavior
10. **Test matrix comment** — MECE matrix at top of file (backend tests)
11. **Multiple assertions per state** — one test verifies all aspects of a single state
12. **Correct mock scope** — mock only external dependencies, not internal modules

For each test file, rate:
- **pass** — file follows all 12 points
- **must-fix** — file violates points 1-5 (structural issues)
- **should-fix** — file violates points 6-11 (quality issues)
- **suggestion** — minor improvements possible

Group by feature area. Focus on the 47 non-generator test files (the 12 generator/template test files are Phase 999.1 scope and should be noted but not deeply audited).
