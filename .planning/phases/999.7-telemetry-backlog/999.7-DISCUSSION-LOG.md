## Error Reporting UX

### Initial Proposal: Error Boundary + GitHub Issue
Rejected by user. Concerns:
- "Copy context" was vague
- Errors aren't always visible to users (server startup, console errors, silent failures)
- GitHub issue creation too heavy for end users
- Wanted simpler mechanism with visual UI
- No email

### Revised Proposal: devMailbox Pattern for Errors
User called this "a beautiful idea."

| Option | Description | Selected |
|--------|-------------|----------|
| /dev/errors route (devMailbox pattern) | devErrors table + auto-capture from 4 error sources + visual dashboard route | ✓ |
| Error boundary + GitHub issue | Report button opens pre-filled GitHub issue | Rejected |
| Error boundary + POST to Convex | Partial — evolved into the devMailbox pattern | |
| Sentry/LogRocket only | External dependency, mandated for all users | Rejected as sole solution |

### Error Visibility: How Maintainer Gets Notified
User asked: "How do I get to know these errors to fix them?"

| Option | Description | Selected |
|--------|-------------|----------|
| Scheduled digest to webhook | Convex cron POSTs error summary on configurable schedule | ✓ |
| Real-time alerts for critical errors | Immediate POST for severe errors | Discussed, deferred |
| Manual `feather report` command | Client runs it manually | Not proposed — user wanted automation |

**User's choice:** Scheduled digest with configurable interval (hourly/daily/weekly)
**Notes:** User specifically wanted client Convex → maintainer's admin Convex as the transport. "Each of these clients' Convex sending it to my overall admin Convex is a great idea." Schedule should be configurable, not just daily — hourly option for active debugging, weekly for stable clients. Consent-based, can be turned off.

### Production Layer 2: Sentry

| Option | Description | Selected |
|--------|-------------|----------|
| Sentry opt-in via DSN | Activates when SENTRY_DSN configured. Not mandated. | ✓ |
| Sentry required | All apps must use Sentry | Rejected |

**User's choice:** Sentry as opt-in Layer 2
**Notes:** User has beta clients with close relationships. Sentry is for when maintainer isn't physically present. Not mandated — "my clients do not have to really do this."

## Privacy

User's response (verbatim summary): "Initial stage, I would like to get everything. No worry about anything private. Don't want API keys or data shapes [to leak]. Can be opt-in. Want sample rows for now. Beta tested with close clients with longtime relationships. Build the right configuration so it can be turned off for someone not known to me."

| Option | Description | Selected |
|--------|-------------|----------|
| Commands + features only | Minimal tracking | Default for public |
| Commands + anonymized schema shape | + field counts and types | Opt-in tier |
| Full (everything except secrets) | + sample rows | Opt-in tier (beta) |

**User's choice:** All three as tiered levels (basic → schema → full)
**Notes:** Three-tier system. Beta clients: `level: full`. Public: `level: basic` default. API keys never collected at any level. All configurable, all can be turned off.
