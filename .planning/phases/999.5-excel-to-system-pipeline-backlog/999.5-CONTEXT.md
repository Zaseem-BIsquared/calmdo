# Phase 999.5: Excel-to-System Pipeline — Context

**Gathered:** 2026-03-30
**Status:** Ready for planning

<domain>
## Phase Boundary

User uploads an Excel file, the system infers schema (entities, fields, types, relationships) via LLM analysis, user confirms through a step-by-step wizard, and the system produces feather.yaml + imports the data into Convex. Schema reconciliation for re-imports (detecting column renames, additions, removals, type changes) is the #1 differentiator — no existing tool handles this well.

</domain>

<decisions>
## Implementation Decisions

### Import Entry Point
- **D-01:** Web upload + CLI. Primary: standalone `/import` route with drag-and-drop zone, progress steps, and full confirmation wizard. Also available as `feather import data.xlsx` CLI command for power users. Both trigger the same pipeline.
- **D-02:** Schema mapping persists in Convex table (primary, web-accessible) with `feather export-mappings` for JSON export (version control / backup).

### Confirmation UX — Step-by-Step Wizard + Dashboard Review
- **D-03:** 3-step wizard for initial review: Step 1 = entities (sheet → entity type with confidence %), Step 2 = fields per entity (column → type + required + relationship), Step 3 = relationships. Back/Next navigation, step indicator for direct jumps.
- **D-04:** After wizard completion, a single-page dashboard review (all entities in tabbed/accordion layout with inline editing) before final "Confirm & Generate" button. Wizard = guided setup; dashboard = review/tweak.
- **D-05:** Inline dropdowns for corrections. Click type cell → dropdown with valid types. Quick, low friction. Matches Tableau Standard.
- **D-06:** Sample data always shown — 2-3 rows from Excel alongside each entity's fields for validation.

### Large Excel Handling (30+ columns)
- **D-07:** LLM groups columns semantically (Personal Info, Financial, Employment, Relationships, Misc). User reviews one group at a time within the Fields step. Groups with low-confidence fields (< 85%) expand first; high-confidence groups collapsed by default.
- **D-08:** Sample data display is togglable: default = inline sample values per field (3 values next to each column), toggle = scrollable full sample rows per group. Consistent toggle pattern across the import UX.

### Re-import & Reconciliation
- **D-09:** Auto-detect re-import from content (sheet names, column patterns, data fingerprints compared against existing mappings in Convex). If similarity > threshold, prompt: "This looks like an update to [entity]. Import as update?" Manual override available: "This is actually a new import" or "Match to a different prior import."
- **D-10:** Reconciliation diff UI is togglable: default = flat change list with action dropdowns per change, toggle = git-style side-by-side diff. Consistent with D-08 toggle pattern.
- **D-11:** Removed columns: prompt per column by default (archive / delete / keep hidden). For large imports, "Archive all & review later" escape hatch. All decisions revisitable.
- **D-12:** Import history as a first-class entity. Each import gets a permalink (`/imports/{id}`) showing the full reconciliation log: columns added, renamed, removed, type changes, data changes, skipped rows. User can revisit and change decisions later. Requires an `imports` table in Convex.

### Data Import Pipeline
- **D-13:** Auto-import after schema confirmation. User confirms → YAML generated → code generated → data imported in dependency order (lookups → masters → transactions). One flow, no extra steps.
- **D-14:** Three-tier error handling for rows that don't fit the confirmed schema:
  - **Green (auto-fixed):** Safe coercions (string "50000" → number, empty optional → null). Done automatically, shown in import report.
  - **Yellow (needs review):** Ambiguous cases. System proposes a default fix, user confirms or overrides. E.g., "Department 'Mktg' doesn't match any lookup — closest: 'Marketing' (89%). Accept?"
  - **Red (unfixable):** No reasonable fix. Skipped. Report shows why + how to fix in Excel.
- **D-15:** All three tiers visible in the import report page (same permalink from D-12).

### Claude's Discretion
- Web framework for the wizard/dashboard UI (existing Radix + TanStack patterns vs dedicated import components)
- Confidence threshold for auto-detect re-import (D-09) — likely 80-90%
- Confidence threshold for column grouping (D-07) — 85% boundary between expand/collapse
- Import report page layout and filtering
- CLI `feather import` argument parsing and flags

</decisions>

<specifics>
## Specific Ideas

- **Gmail contact import as UX reference**: Import history should work like Gmail's contact import — a link that lets you go back and see what happened at a specific import event
- **Tableau as the guiding principle**: Infer first, smart defaults, never ask what you can figure out. Import should feel like dragging an Excel into Tableau — it just works, you tweak if needed
- **Toggle pattern**: Both sample data views and diff views should be togglable (compact default + detailed alternate). Consistent UX pattern across the entire import experience.

</specifics>

<deferred>
## Deferred Ideas

- **LLM-generated domain-realistic seed data** — When developer uses Architect Conversation (999.3) without Excel, LLM generates 3-5 realistic rows per entity based on domain description. Belongs in Phase 999.3 scope, not here. Import pipeline (dependency ordering, error handling, import history) is reusable.
- **Flat data → entity splitting** — What happens with one big denormalized sheet? LLM entity classification from architecture doc handles this, but the UX for confirming entity splits from flat data is unresolved. Defer to research phase.
- **File format expansion** — CSV, Google Sheets, Airtable exports. Start with xlsx (SheetJS) per ROADMAP. Add formats based on user demand.
- **Performance / large file handling** — 100MB+ files, 50k+ rows, streaming vs memory. Defer to research/planning — likely needs chunked import with progress.

</deferred>
