# Phase 999.7: Telemetry - Context

**Gathered:** 2026-03-30
**Status:** Ready for planning

<domain>
## Phase Boundary

Understand what users build and surface errors proactively. Three sub-systems: CLI telemetry (generator usage tracking), error capture and reporting (devMailbox pattern + scheduled digests), and maintainer visibility (`feather stats` + admin Convex aggregation).

</domain>

<decisions>
## Implementation Decisions

### CLI Telemetry
- **D-01:** Every `feather` CLI command appends to `.feather/telemetry.json` locally. Tracks: command name, feature/bundle name, success/failure, duration, timestamp.
- **D-02:** Optional remote endpoint configured in `feather.yaml` (`telemetry.remote`). When configured, telemetry events POST to the endpoint. When not configured, local-only.
- **D-03:** Opt-out via `telemetry.enabled: false` in `feather.yaml` or `FEATHER_TELEMETRY=false` env var.

### Error Capture — devMailbox Pattern
- **D-04:** `devErrors` Convex table stores every error with full context. Same pattern as existing `devEmails` table / `/dev/mailbox` route.
- **D-05:** `/dev/errors` route inside the dashboard layout shows all captured errors. Visual dashboard — error list with expandable details (stack trace, route, function name, args, browser info, timestamp). Includes "Clear All" action.
- **D-06:** Auto-capture from 4 error sources:
  - **Frontend crash:** React ErrorBoundary component catches unhandled errors → mutation stores to `devErrors`
  - **Backend error:** Convex function error wrapper/middleware → stores to `devErrors`
  - **Silent failure:** ConvexError pattern → stores to `devErrors`
  - **Startup error:** CLI captures → stores to local `.feather/errors.json` (separate from Convex since backend may not be running)
- **D-07:** `/dev/errors` route gated by `DEV_ERRORS` env var (same pattern as `DEV_MAILBOX`). Defaults to enabled in dev.

### Error Visibility — Scheduled Digest
- **D-08:** Convex scheduled function (cron) runs on configurable interval, collects error summary from `devErrors` table, and POSTs to maintainer's endpoint.
- **D-09:** Client Convex → maintainer's admin Convex instance. The admin instance aggregates error digests from all client deployments into one view. Natural fit since both sides run Convex.
- **D-10:** Schedule configurable: `hourly | daily | weekly`. Default: `daily`.
- **D-11:** Digest contents: error count since last report, top errors by frequency (message + stack trace + route/function), environment info (feature versions, browser breakdown). No user PII, no data values.
- **D-12:** Consent-based. `telemetry.errorDigest.enabled` in `feather.yaml`. Can be turned off by the client. Must be explicitly configured (endpoint URL) to activate.

### Production Error Reporting — Sentry Opt-In (Layer 2)
- **D-13:** Sentry integration as opt-in Layer 2. Activates when DSN is configured via `feather.yaml` (`errorReporting.sentry.dsn`) or `SENTRY_DSN` env var.
- **D-14:** When Sentry is configured, errors auto-report to Sentry in addition to `devErrors` table. When not configured, only Layer 1 (devErrors + digest) operates.
- **D-15:** Sentry is NOT mandated — clients don't have to use it. It's there for production monitoring when the maintainer isn't in the room.

### Maintainer Dashboard
- **D-16:** `feather stats` CLI command reads local `.feather/telemetry.json`. Shows: feature inventory, generator usage counts, error rates, most-generated features. Terminal output, no web UI needed for CLI stats.

### Privacy — Tiered Granularity
- **D-17:** Three telemetry tiers, configurable in `feather.yaml`:
  - **basic** (default for public users): Commands run, features generated/added, success/fail, duration
  - **schema** (opt-in): + field names, types, relationships, schema shape
  - **full** (opt-in, beta clients): + sample data rows
- **D-18:** Never collected at any tier: API keys, environment variables, secrets, auth tokens.
- **D-19:** Beta clients (close relationships) run `level: full`. Public users get `level: basic` by default. Configuration: `telemetry.level: basic | schema | full`.

### Configuration Surface
- **D-20:** All telemetry configured in `feather.yaml` under a `telemetry` section:

```yaml
telemetry:
  enabled: true            # master switch (default: true)
  level: basic             # basic | schema | full
  remote: https://...      # optional CLI telemetry endpoint
  errorDigest:
    enabled: true          # consent toggle
    schedule: daily        # hourly | daily | weekly
    endpoint: https://admin.convex.site/api/errors
```

### Claude's Discretion
- Error boundary component design (full-page vs inline, retry button behavior)
- Convex error wrapper/middleware implementation approach
- `devErrors` table schema (fields, indexes)
- Admin Convex instance setup (receiving endpoint, aggregation queries, admin UI)
- Sentry SDK integration specifics (which package, initialization pattern)
- `feather stats` output formatting and time range options
- Whether startup errors sync to `devErrors` once backend becomes available

</decisions>

<specifics>
## Specific Ideas

- devMailbox pattern is the proven model — same table + route + env var gating approach
- Client Convex → admin Convex is the natural aggregation path since both sides already run Convex
- Beta clients get `level: full` telemetry (close relationship, data access already established)
- Public users get `level: basic` with clear opt-out
- Error boundary is a gap (F1-10 in VISION.md was never built) — this phase fills it
- Sentry is Layer 2 for when maintainer isn't in the room — not mandated for anyone

</specifics>

<deferred>
## Deferred Ideas

- **Admin Convex dashboard UI** — visual web dashboard aggregating all client digests. For now, the data flows to admin Convex; dashboard UI is a future phase.
- **Error severity classification** — auto-classify errors as critical/warning/info based on type and frequency. Currently all errors are equal.
- **Client-side error search/filter** — filtering `/dev/errors` by route, function, time range. Basic list view first.
- **Sentry session replay integration** — if using Sentry, session replay gives full context. Expensive, defer to when clients request it.

</deferred>
