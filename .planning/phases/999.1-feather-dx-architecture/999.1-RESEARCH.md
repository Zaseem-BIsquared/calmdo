**Researched:** 2026-03-28
**Mode:** Synthesis of existing design research + technical gap analysis
**Sources:** 10 design documents in `.planning/design/`, codebase analysis, framework comparisons

---

## Standard Stack

### YAML Processing
- **Use `yaml` (npm)** — already in project deps. Handles parsing, serialization, schema validation with custom types. No additional dependency needed.
- **Use Zod v4 for YAML schema validation** — already the project's validation standard. Define a Zod schema for `feather.yaml` format, validate parsed YAML against it. Pattern: `yaml.parse()` -> `featherYamlSchema.parse()`.

### AST-Based Code Wiring
- **Use `ts-morph`** for all AST manipulation (schema.ts wiring, nav.ts appending, i18n.ts namespace registration, errors.ts group insertion).
  - Why ts-morph: TypeScript-first API, understands TS project structure, handles imports/exports natively, simpler API than jscodeshift for TS-specific transforms.
  - Why NOT jscodeshift: React/JSX-focused, weaker TS support, Facebook-centric community. ts-morph is the standard for TS tooling (used by NestJS CLI, Angular CLI, Nx generators).
  - Why NOT custom: The 3 existing append-only files (nav.ts, errors.ts, i18n.ts) and schema.ts all need reliable AST manipulation. Hand-rolling is the #1 source of generator bugs.
- **Install:** `npm install ts-morph` (runtime dep for generator CLI).

### Code Generation (Templates)
- **Keep Plop.js + Handlebars** for file scaffolding — 26 templates already exist and work. Evolution path: Plop generates into `generated/` directory instead of feature root.
- **Extend with AST wiring** as post-generation step — Plop creates files, ts-morph wires them into shared files. Two-phase pipeline: scaffold + wire.

### CLI Framework
- **Use `commander`** for the `feather` CLI. Lightweight, zero-config, TypeScript-friendly. Standard for Node.js CLIs (used by Vite, Next.js, Create React App).
- **Alternative considered:** `yargs` — heavier, more features than needed. `oclif` — enterprise-grade but too much framework for this use case.

### Excel Processing
- **Use `xlsx` (SheetJS)** for Excel reading/writing. The de facto standard. Handles .xlsx, .xls, .csv.
- **Use `string-similarity` or `fastest-levenshtein`** for fuzzy column rename detection (Jaro-Winkler / Levenshtein distance). Lightweight, no ML dependency.

### LLM Integration
- **Use `@anthropic-ai/sdk`** for the architect conversation. Direct API calls, streaming support, tool use for structured entity extraction.
- **Conversation state:** JSON files in `.feather/conversations/` — resumable, inspectable, no database needed.

## Architecture Patterns

### 1. Two-Phase Generator Pipeline (Scaffold + Wire)

```
feather generate <feature-name>
  Phase 1: Plop.js renders Handlebars templates → generated/ files
  Phase 2: ts-morph modifies shared files (schema.ts, nav.ts, i18n.ts, errors.ts, routes)
```

This matches the current Plop-based approach but adds reliable wiring. The generator stress tests found 6+ bugs caused by manual wiring — AST-based wiring eliminates this class of errors.

### 2. Generated/Custom Directory Split

```
src/
  generated/{feature}/     # Plop output — regeneratable, never hand-edit
    components/
    hooks/
    index.ts (barrel)
  custom/{feature}/        # User code — never touched by generator
    overrides/             # YAML-declared overrides (D-08)
    extensions/            # Custom business logic
convex/
  generated/{feature}/     # Backend mutations, queries, schema fragment
  custom/{feature}/        # Custom backend logic (beforeSave hooks, etc.)
```

**Key constraint:** Generated code NEVER imports from custom (D-07). Custom imports from generated and wraps/extends. This is the Frappe pattern: hooks.py extends DocType behavior without modifying the DocType definition.

### 3. Feature YAML as Single Source of Truth

Each feature has one YAML file that declares everything the generator needs:

```yaml
# src/features/tasks/feather.yaml (evolved from tasks.gen.yaml)
name: tasks
label: Task

# Schema (Dimension 0: baseline)
fields: { ... }
identity: { type: expression, format: "TASK-{###}" }

# Dimension 1: Access Control
access:
  scope: custom
  permissions: { ... }

# Dimension 2: Status Flow
statusFlow:
  field: status
  transitions: { ... }

# Dimension 3: Side Effects
hooks:
  beforeSave: custom/tasks/hooks.ts#beforeSave
  onStatusChange: custom/tasks/hooks.ts#onStatusChange

# Dimension 4: Derived Data
derivedData:
  subtaskCount: { type: count, source: subtasks }
  totalTimeLogged: { type: sum, source: workLogs, field: minutes }

# Dimension 5: Custom Views
views: { ... }

# Dimension 6: Schedules (when applicable)
schedules:
  overdueCheck: { cron: "0 9 * * *", action: markOverdue }

# Dimension 7: Integrations (when applicable)
integrations: { ... }

# Dimension 8: Identity (handled above in identity field)

# Relationships
relationships: { ... }
children:
  lineItems: { ... }  # Tight parent-child (D-02)

# Detail View Composition (D-04)
detailView:
  relatedRecords:
    - { source: subtasks, display: checklist }
    - { source: workLogs, display: timeline }
    - { source: activityLogs, display: timeline }

# Override pointers (D-08)
overrides:
  form: custom/tasks/CustomForm.tsx
```

### 5. Cross-Entity UI Generation Pattern

The #1 gap identified by stress tests. Master-detail views require:

```yaml
# In parent feature YAML (D-04)
detailView:
  layout: tabs  # or: panels, accordion
  relatedRecords:
    - source: lineItems       # child entity
      display: table           # table | checklist | timeline | cards
      inline: true             # create/edit inline vs navigate away
      columns: [name, qty, price, total]
    - source: activityLogs
      display: timeline
      readonly: true
```

Generator produces a `DetailPage` component that:
1. Renders the parent entity fields
2. Renders related record panels based on `relatedRecords` config
3. Each panel uses the appropriate display component (TableView, ChecklistView, TimelineView)
4. Inline editing panels get embedded forms

### 6. Safe Upstream Updates (D-11, D-12)

```
feather update
  1. Regenerate all generated/ files from latest templates
  2. Run AST codemods on custom/ files if breaking API changes
  3. Run typecheck to verify custom → generated imports still work
  4. Report: files regenerated, codemods applied, errors found
```

Codemods shipped per release in `node_modules/feather/codemods/{version}/`. Pattern borrowed from Next.js (`npx @next/codemod`).

## Don't Hand-Roll

| Problem | Use Instead | Why |
|---------|-------------|-----|
| AST manipulation | ts-morph | Hand-rolling TS AST transforms is the #1 source of wiring bugs (6+ found in stress tests) |
| YAML schema validation | Zod schema for YAML | Already the project standard; don't create a parallel validation system |
| File wiring (imports, exports, array appends) | ts-morph `SourceFile.addImportDeclaration()`, `ArrayLiteralExpression.addElement()` | Regex-based file modification breaks on edge cases (comments, multi-line, trailing commas) |
| CLI argument parsing | commander | Don't parse process.argv manually |
| Excel column matching | string-similarity library | Don't implement Jaro-Winkler from scratch |
| Template rendering | Handlebars (via Plop) | 26 templates already working; don't switch engines |
| State machine validation | Simple lookup table | Don't pull in xstate for YAML-declared status transitions |

## Common Pitfalls

### 1. Generated/Custom Boundary Leaks
**Risk:** Developers edit generated files, lose changes on update.
**Mitigation:** Generated files include a header comment: `// @generated — DO NOT EDIT. Customize in custom/{feature}/`. The `feather update` command warns if generated files have uncommitted changes.

### 2. YAML Schema Bloat
**Risk:** The YAML format tries to express everything and becomes as complex as the code it generates.
**Mitigation:** Apply D-13 (Tableau principle): sensible defaults for everything, progressive disclosure. A minimal YAML (just `name` and `fields`) must produce a working feature. Each dimension is opt-in.

### 3. Circular Import Between Generated and Custom
**Risk:** Custom code imports from generated (correct), but generated accidentally imports from custom.
**Mitigation:** Lint rule: `no-restricted-imports` in generated/ that blocks `../custom/` imports. Enforced at generation time (templates never reference custom paths) and at build time (ESLint rule).

### 4. AST Wiring Breaks on Unexpected File Formats
**Risk:** ts-morph assumes specific file structure (e.g., `navItems` is an exported array literal). User reformats file, breaks wiring.
**Mitigation:** Use marker comments (`// feather:nav-items-start` / `// feather:nav-items-end`) as anchors. ts-morph finds markers first, falls back to AST structure if markers present.

### 5. Cross-Entity UI Template Explosion
**Risk:** Every combination of parent display + child display + inline editing creates a new template variant.
**Mitigation:** Compose from primitives, not templates. Generate a `DetailPage` shell that imports `<RelatedRecordPanel display={config.display} />` — the panel component handles table/timeline/checklist switching at runtime via props, not template-time.

### 6. Schema.ts Merge Conflicts
**Risk:** Multiple features adding to `convex/schema.ts` simultaneously causes merge conflicts.
**Mitigation:** Each feature generates a schema fragment file (`convex/generated/{feature}/schema.fragment.ts`). A single `convex/schema.ts` aggregator imports and spreads all fragments. No two features edit the same file.

### 7. YAML Validation Errors are Opaque
**Risk:** User gets `ZodError: invalid_type at .fields.status.transitions.todo[0]` — meaningless without context.
**Mitigation:** Custom error formatter that maps Zod paths to human-readable messages with YAML line numbers. Use `yaml` library's `lineCounter` option to track source positions.

### 8. Existing Feature Migration is All-or-Nothing
**Risk:** Migrating tasks/projects/etc from current structure to generated/custom split requires moving all features at once.
**Mitigation:** Support hybrid mode during migration — features without YAML declarations continue working as-is. The generator only manages features that have a `feather.yaml` file. Gradual adoption.

## Code Examples

### ts-morph: Wiring a New Feature into schema.ts

```typescript
import { Project, SyntaxKind } from "ts-morph";

function wireSchemaFragment(featureName: string, projectRoot: string) {
  const project = new Project({ tsConfigFilePath: `${projectRoot}/convex/tsconfig.json` });
  const schemaFile = project.getSourceFileOrThrow("convex/schema.ts");

  // Add import
  schemaFile.addImportDeclaration({
    moduleSpecifier: `./generated/${featureName}/schema.fragment`,
    namedImports: [`${featureName}Table`],
  });

  // Find defineSchema call and add table
  const defineSchemaCall = schemaFile.getDescendantsOfKind(SyntaxKind.CallExpression)
    .find(c => c.getExpression().getText() === "defineSchema");

  if (defineSchemaCall) {
    const arg = defineSchemaCall.getArguments()[0];
    if (arg?.isKind(SyntaxKind.ObjectLiteralExpression)) {
      arg.addPropertyAssignment({
        name: featureName,
        initializer: `${featureName}Table`,
      });
    }
  }

  schemaFile.saveSync();
}
```

### ts-morph: Appending to nav.ts

```typescript
function wireNavItem(featureName: string, label: string, icon: string, path: string) {
  const navFile = project.getSourceFileOrThrow("src/shared/nav.ts");

  // Add icon import
  const lucideImport = navFile.getImportDeclaration(m =>
    m.getModuleSpecifierValue() === "lucide-react"
  );
  lucideImport?.addNamedImport(icon);

  // Find navItems array and append
  const navItems = navFile.getVariableDeclarationOrThrow("navItems");
  const initializer = navItems.getInitializerIfKindOrThrow(SyntaxKind.ArrayLiteralExpression);

  initializer.addElement(`{
    label: "${label}",
    to: "${path}",
    icon: ${icon},
  }`);

  navFile.saveSync();
}
```

### Feature YAML Validation with Zod

```typescript
import { z } from "zod/v4";
import { parse as parseYaml } from "yaml";

const fieldSchema = z.object({
  type: z.enum(["string", "text", "number", "boolean", "enum", "date", "reference"]),
  required: z.boolean().default(false),
  max: z.number().optional(),
  default: z.union([z.string(), z.number(), z.boolean()]).optional(),
  values: z.array(z.string()).optional(),  // for enum type
  filterable: z.boolean().default(false),
  sortable: z.boolean().default(true),
  transitions: z.record(z.string(), z.array(z.string())).optional(),
});

const featureYamlSchema = z.object({
  name: z.string(),
  label: z.string(),
  labelPlural: z.string().optional(),
  fields: z.record(z.string(), fieldSchema),
  identity: z.object({
    type: z.enum(["auto-increment", "uuid", "expression"]),
    format: z.string().optional(),
  }).default({ type: "auto-increment" }),
  access: z.object({
    scope: z.enum(["owner", "shared", "public", "custom"]).default("owner"),
    permissions: z.record(z.string(), z.string()).optional(),
  }).optional(),
  // ... other dimensions
});

function validateFeatureYaml(yamlContent: string) {
  const parsed = parseYaml(yamlContent);
  return featureYamlSchema.parse(parsed);
}
```

### Two-Phase Generator Pipeline

```typescript
// Phase 1: Scaffold with Plop
async function scaffoldFeature(config: FeatureConfig) {
  const plop = await nodePlop("./plopfile.js");
  const generator = plop.getGenerator("feature");
  await generator.runActions({ ...config, outputDir: "generated" });
}

// Phase 2: Wire with ts-morph
async function wireFeature(featureName: string) {
  wireSchemaFragment(featureName, process.cwd());
  wireNavItem(featureName, config.label, config.icon, `/${featureName}`);
  wireI18nNamespace(featureName);
  wireErrorConstants(featureName);
}

// Combined pipeline
async function generateFeature(yamlPath: string) {
  const config = validateFeatureYaml(readFileSync(yamlPath, "utf8"));
  await scaffoldFeature(config);
  await wireFeature(config.name);
  console.log(`Feature ${config.name} generated and wired.`);
}
```

## Validation Architecture

### Testing the Generator Pipeline

The generator itself needs tests:

1. **YAML schema validation tests** — valid/invalid YAML inputs, edge cases (empty fields, missing required)
2. **Template output tests** — snapshot tests for each template with representative inputs
3. **AST wiring tests** — verify ts-morph operations produce correct file modifications
4. **End-to-end generator test** — run generator with test YAML, verify all files created, all wiring complete, typecheck passes

### Testing Generated Features

Generated features should pass:
1. **TypeScript typecheck** — generated code must compile
2. **Vitest unit tests** — generated test files must pass
3. **100% coverage** — generated code must meet project coverage requirements
4. **ESLint** — no lint violations in generated code

### Migration Testing

For existing features migrating to generated/custom:
1. **Behavior preservation** — migrated feature produces identical API surface
2. **Test preservation** — existing tests continue passing after migration
3. **No regression** — full test suite passes after each feature migration

## Open Technical Questions

### 1. Schema Fragment Aggregation Pattern
Two approaches for `convex/schema.ts`:
- **Option A:** Import fragments and spread: `defineSchema({ ...authTables, ...tasksTable, ...projectsTable })`
- **Option B:** Schema fragments export table definitions, aggregator script generates schema.ts

Recommendation: **Option A** — simpler, works with ts-morph, matches current `authTables` spread pattern.

### 2. Route Generation Strategy
TanStack Router uses file-based routing. Generated routes go to `src/routes/_app/_auth/{feature}.tsx`. Currently Plop creates these directly. With generated/custom split:
- Routes stay in `src/routes/` (not in `generated/`) because TanStack Router requires specific file paths
- Route files are thin wrappers (< 20 lines) importing from `generated/` — regeneratable
- Route content IS generated code, just not in the generated/ directory

### 3. Handlebars Template Evolution
Current templates output to feature root. Need to retarget to `generated/{feature}/`. This is a Plop config change (destination path), not template content change. Existing templates work as-is with new output directory.

### 4. Custom Code Discovery
When generating, need to detect existing custom code to avoid conflicts:
- Check `custom/{feature}/` directory exists
- Read override declarations from feature YAML
- Skip generating files that have declared overrides

## RESEARCH COMPLETE

**Key findings:**
1. The 8-dimension framework is well-researched and validated against 5 real systems (75-90% coverage)
2. ts-morph is the clear choice for AST wiring — eliminates the #1 class of generator bugs
3. Cross-entity UI generation (master-detail views) is the highest-impact gap to close
4. The YAML schema format is the foundation — everything else depends on it being right
5. Hybrid mode (existing features + generated features coexist) is critical for adoption

**Recommended implementation order:**
1. Design and implement the full `feather.yaml` schema format with Zod validation
2. Implement ts-morph AST wiring layer (replace manual file modifications)
3. Retarget Plop templates to generated/ directory structure
4. Build cross-entity UI generation (master-detail, related records panels)
5. Build `feather` CLI wrapper
6. Add Excel import pipeline
7. Add LLM architect conversation
8. Add telemetry
