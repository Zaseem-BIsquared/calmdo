# Plan 07-03: Frontend Feature Test Rewrite — MECE State Decomposition

## Objective

Rewrite all 7 active frontend feature test files to follow the feather-testing-convex testing philosophy: MECE state decomposition (loading → empty → with data → error), integration-first approach, philosophy-compliant naming, and proper async patterns (`findBy*` queries instead of `waitFor(() => getBy*(...))` wrappers).

## Why This Matters

Frontend tests are where the philosophy's integration-first approach has the biggest impact. Currently, tests verify features but lack systematic state coverage — loading spinners aren't tested, empty states are sometimes missing, and the code uses `waitFor(() => { expect(screen.getBy*(...)) })` instead of the cleaner `findBy*` pattern. Additionally, route `beforeLoad` and navItem tests are duplicated across feature files when they belong in centralized locations.

## must_haves
- [ ] All 7 frontend test files have MECE test matrix comments listing every component state
- [ ] Each component state has exactly one test (no overlap, no gaps)
- [ ] Integration tests use `seed()` or mutation API for data setup (not raw `ctx.db.insert()`)
- [ ] Mock tests used ONLY for loading spinners and error states
- [ ] `findBy*` queries used for async data instead of `waitFor(() => getBy*(...))` — applies to `findByText`, `findByRole`, `findByPlaceholderText`, `findByDisplayValue`, etc. Reserve `waitFor` only for non-query assertions (className checks, backend state)
- [ ] Route `beforeLoad` tests removed from feature files (they belong in route tests, or kept if they're the only location)
- [ ] Nav item tests removed from feature files (centralized in `src/shared/nav.test.ts` by Plan 02)
- [ ] Test names follow `"verb what-user-sees [condition]"` convention
- [ ] `npm test` passes with 100% coverage
