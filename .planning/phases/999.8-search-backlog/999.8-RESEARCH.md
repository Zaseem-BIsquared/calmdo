**Goal:** Users can text-search across task titles and project names with filter controls

## 1. Convex Search Index Architecture

### How searchIndex Works

Convex provides native full-text search via `.searchIndex()` declarations on table definitions in `schema.ts`:

```typescript
tasks: defineTable({ ... })
  .searchIndex("search_title", {
    searchField: "title",           // ONE field gets full-text search
    filterFields: ["status", "assigneeId", "projectId", "priority"],
  })
```

**Query usage:**
```typescript
const results = await ctx.db
  .query("tasks")
  .withSearchIndex("search_title", (q) =>
    q.search("title", searchTerm).eq("status", "todo")
  )
  .take(50);
```

**Key constraints (from Phase 6 research):**
- Only ONE `searchField` per search index (the field that gets full-text search)
- Multiple `filterFields` for equality filtering alongside search
- Results are ranked by relevance, not creation time
- `.take(n)` is required — no `.collect()` on search queries
- Prefix matching is supported (typing "tas" matches "tasks")
- Search indexes are separate from regular indexes and can coexist

### Search Indexes Needed

**Tasks — `search_title`:**
```typescript
.searchIndex("search_title", {
  searchField: "title",
  filterFields: ["status", "assigneeId", "projectId", "priority"],
})
```

**Projects — `search_name`:**
```typescript
.searchIndex("search_name", {
  searchField: "name",
  filterFields: ["status"],
})
```

**Decision: Title-only search for tasks (not description).** Searching both title and description would require either a second search index with separate query merging, or a denormalized `searchText` field. Title-only is pragmatic for now. Description search is a natural future extension.

## 2. Current Query Architecture

### Tasks Queries (`convex/tasks/queries.ts`)

Current queries:
- `myTasks` — uses `by_assignee` index, returns tasks assigned to current user
- `teamPool` — uses `by_visibility` index, returns shared unassigned tasks
- `getById` — direct document lookup
- `listUsers` — returns all users (for assignee dropdowns)

**For filtered task lists**, we need new queries that:
1. Accept optional filter params (status, priority, projectId, assigneeId)
2. When a search term is present, use the search index
3. When no search term, use regular indexes with client-side filtering for combinations

**Challenge:** Convex doesn't support compound queries across multiple indexes. A query using `by_assignee_status` can filter by assignee + status but not also by project. Options:
- **Option A: Search index for everything** — When ANY filter is active, use `withSearchIndex` with empty search term + filterFields. However, empty search term may not work with search indexes.
- **Option B: Fetch + client-side filter** — Fetch with the most selective index, then filter remaining criteria in JS. For a task management app with < 10,000 tasks, this is fine.
- **Option C: Multiple queries** — One query per filter dimension, intersect results.

**Recommendation: Option B** — fetch tasks using the best available index, then filter the rest client-side. The data volume doesn't justify complex query strategies. When a search term is present, switch to the search index.

### Projects Queries (`convex/projects/queries.ts`)

Current queries:
- `list` — already accepts optional `status` filter, uses `by_status` index. Already works for SRCH-05.
- `getWithTasks` — returns project with all tasks + status summary.

Projects only need a search query variant that uses the `search_name` index.

### URL Search Params (CONTEXT.md D-08)

Filters persist via URL query params: `/dashboard/tasks?status=in_progress&assignee=me`

TanStack Router supports this natively via `validateSearch` in route definitions (already used in projects route). Pattern:

```typescript
export const Route = createFileRoute("/_app/_auth/dashboard/_layout/tasks")({
  validateSearch: (search: Record<string, unknown>) => ({
    status: search.status as string | undefined,
    priority: search.priority as string | undefined,
    assignee: search.assignee as string | undefined,
    project: search.project as string | undefined,
  }),
});
```

## 4. Global Search — Frontend Architecture

### Search Box in Header (CONTEXT.md D-03, D-04)

Per decisions: Global search box in header/nav bar. Always visible. Searches across all entities (tasks + projects). Results appear as a dropdown overlay while typing.

**Component structure:**
```
GlobalSearchBox (in Navigation component)
├── SearchInput (with Cmd+K shortcut, debounced 200-300ms)
└── SearchResultsDropdown
    ├── TaskResults section
    └── ProjectResults section
```

**Integration point:** The `Navigation` component (`src/features/dashboard/components/Navigation.tsx`) renders the nav bar. The search box fits between the left nav items and the right user menu area.

### Search Flow

1. User clicks search box or presses Cmd+K
2. Input is debounced (250ms)
3. Two parallel Convex queries fire: `tasks.queries.search` + `projects.queries.search`
4. Results appear in dropdown, grouped by entity type
5. Clicking a result navigates to the entity (task detail panel or project detail page)
6. Pressing Escape or clicking outside closes the dropdown

### Backend Search Queries

```typescript
// convex/tasks/queries.ts
export const search = query({
  args: { searchTerm: v.string(), limit: v.optional(v.number()) },
  handler: async (ctx, args) => {
    const userId = await auth.getUserId(ctx);
    if (!userId) return [];
    return ctx.db
      .query("tasks")
      .withSearchIndex("search_title", (q) => q.search("title", args.searchTerm))
      .take(args.limit ?? 10);
  },
});

// convex/projects/queries.ts
export const search = query({
  args: { searchTerm: v.string(), limit: v.optional(v.number()) },
  handler: async (ctx, args) => {
    const userId = await auth.getUserId(ctx);
    if (!userId) return [];
    return ctx.db
      .query("projects")
      .withSearchIndex("search_name", (q) => q.search("name", args.searchTerm))
      .take(args.limit ?? 10);
  },
});
```

- `Input` — for search text input
- `DropdownMenu` — already used extensively, could serve for search results overlay
- `Popover` — better fit for search results dropdown (stays open while interacting)

### Test Patterns

- Backend tests use `test` fixture from `@cvx/test.setup` with `mutation`, `query`, `auth` helpers
- Frontend tests use `renderWithRouter` from `@/test-helpers` with `ConvexTestClient`
- 100% coverage required, enforced by pre-commit hook

### Note on Search Index Testing

Convex test helper (`feather-testing-convex`) runs an in-memory backend. Search indexes may or may not be supported in the test environment. If search index queries fail in tests, the fallback is to mock the search results or test with regular queries and validate the search index is declared correctly in schema.

## Validation Architecture

### Dimension 1: Functional Correctness
- Search returns matching results for partial and full terms
- Each filter narrows results correctly (status, priority, assignee, project)
- Combined filters work (status + priority simultaneously)
- Global search returns both tasks and projects

### Dimension 2: Data Integrity
- Search indexes declared on correct fields
- Filter values validated against schema enums
- URL params sanitized via `validateSearch`

### Dimension 3: Performance
- Search queries use indexes (not table scans)
- Debounced search input (250ms) prevents excessive queries
- Results limited via `.take()` on search queries
- Filter changes don't cause full page re-renders

### Dimension 4: Edge Cases
- Empty search term: no results shown (not all results)
- No matching results: empty state message
- Very long search terms: truncated or rejected
- Special characters in search: handled gracefully
- Filter with zero results: clear empty state

### Dimension 5: Integration
- Search box visible on all dashboard pages
- Cmd+K shortcut doesn't conflict with browser shortcuts
- Filter state survives navigation and page refresh (URL params)

### Dimension 6: Accessibility
- Search input has proper label/aria attributes
- Filter dropdowns keyboard-navigable
- Search results navigable via arrow keys
- Escape key closes search dropdown

### Dimension 7: Regression
- Existing task queries (myTasks, teamPool) unchanged
- Existing project queries (list, getWithTasks) unchanged
- Navigation component layout not broken by search box addition
- Test suite passes at 100% coverage
