# Phase 999.6: Bundle Distribution - Context

**Gathered:** 2026-03-30
**Status:** Ready for planning

<domain>
## Phase Boundary

Curated multi-feature bundles installable in one command, with local-first resolution and GitHub registry fallback for remote distribution. Covers: bundle manifest format, CLI commands (`feather add/remove/list/update`), dependency resolution via topological sort, and registry-ready format for remote install and updates.

</domain>

<decisions>
## Implementation Decisions

### Terminology
- **D-01:** Two concepts only — **features** (atomic installable unit) and **bundles** (named collection of features). "Example" is a documentation label, not a CLI concept. `templates/examples/` renamed to `templates/features/`.

### Bundle Manifest Format
- **D-02:** Bundle manifest is `bundle.json` containing feature list + metadata only: `{ name, label, description, complexity, features: [...] }`. No `installOrder` — derived from `feather.yaml` dependency graph. No cross-entity wiring — that stays in individual `feather.yaml` `detailView.relatedRecords` declarations.
- **D-03:** `templates/bundles/` directory, separate from `templates/features/`. Auto-detect in `feather add` checks `bundles/` first, then `features/`.

### Distribution Channel (Model C — Hybrid)
- **D-04:** Local-first with GitHub fallback. `feather add <name>` checks local `templates/` first. If not found, fetches from registry URL configured in `feather.yaml` (`registry.url`). Downloaded bundles/features cached locally in `templates/` for offline use.
- **D-05:** Registry is the GitHub repo itself — features and bundles served as raw files from the repo. No custom infrastructure. Works with private repos (token auth). Versioning via branches/tags in registry config.
- **D-06:** `bundle.json` format is registry-compatible — identical whether read from filesystem or fetched from URL. No format changes needed when adding remote support.

### Command UX
- **D-07:** Simple verbs, no double-dash flags. Full command set:
  - `feather add <name>` — auto-detect feature or bundle, install. Checks `bundles/` first, then `features/`.
  - `feather remove <name>` — auto-detect feature or bundle, uninstall.
  - `feather list` — show installed features, available features, and available bundles.
  - `feather update` — full refresh: sync templates from registry + regenerate all generated/ files.
  - `feather update <name>` — targeted refresh for specific feature(s).
- **D-08:** `feather update` without registry configured regenerates from local templates only (backward compatible with current behavior). With registry configured, syncs first then regenerates.

### Dependency Resolution
- **D-09:** Topological sort derived from `belongs_to` declarations in each feature's `feather.yaml`. Builds a directed acyclic graph, produces install order where every dependency is installed before its dependents. ~20 lines of code, no explicit `installOrder` in `bundle.json` (can't get out of sync).
- **D-10:** Already-installed features get updated (overwrite generated/, preserve custom/), missing features get installed. No skipping — ensures all features in a bundle are at consistent versions.
- **D-11:** Missing dependency outside the bundle = error with clear message: "subtasks requires tasks, which is not installed and not in this bundle."

### Claude's Discretion
- Registry URL format and auth token handling for private repos
- Cache invalidation strategy (timestamp-based, etag, or always-fetch)
- `feather list --remote` behavior (whether to show registry contents vs local-only default)
- Error UX for network failures during remote fetch (fallback to local cache?)
- Whether `feather update` shows a diff/summary before regenerating or just does it

</decisions>

<specifics>
## Specific Ideas

- shadcn v3 registry model is the closest reference — static JSON served from GitHub, CLI fetches and copies files, namespaced registries for multiple sources
- "Least painful" distribution for a team: GitHub repo IS the registry, no infrastructure to maintain
- Client workflow: `feather start project` (Day 1) → `feather add <bundle>` (add features) → `feather update` (get fixes/updates)
- The format should be identical local and remote — `bundle.json` works whether on filesystem or fetched from URL

</specifics>

<deferred>
## Deferred Ideas

- **Registry authentication for private repos** — token-based auth for `registry.url` pointing to private GitHub repos. Design when first private client deployment happens.
- **`feather list --remote`** — show what's available on the registry even if not cached locally. Useful for discovery but requires network.
- **Versioning/pinning** — `registry.ref: v2.1` to pin to a specific branch/tag. Important for production stability but not needed for v1 of the distribution system.
- **Bundle marketplace / directory** — community-contributed bundles beyond the core team's offerings. Way beyond current scope.

</deferred>
