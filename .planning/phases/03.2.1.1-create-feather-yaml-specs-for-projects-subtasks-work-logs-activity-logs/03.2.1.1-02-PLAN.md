# Plan 03.2.1.1-02: Example App feather.yaml Specs + Cleanup

## Objective

Convert the 3 example app gen.yaml files to complete feather.yaml specs (todos, tickets, contacts), update the root feather.yaml to list all 8 entities, then delete all .gen.yaml and .resolved.yaml files. After this plan, `feather.yaml` is the single spec format across the codebase — no `.gen.yaml` files remain.

## Why This Matters

Example apps serve as reference implementations showing different levels of feature complexity (simple toggle, status+priority+assignable, contact management with table views). Complete feather.yaml specs make them useful as test fixtures for generator development and as templates for developers building new features.

## must_haves

- [ ] todos/feather.yaml created as complete spec (simple toggle feature)
- [ ] tickets/feather.yaml created as complete spec (status+priority+assignable feature)
- [ ] contacts/feather.yaml created as complete spec (table-oriented CRM feature)
- [ ] Root feather.yaml updated with all 8 features listed
- [ ] All .gen.yaml files deleted (8 files across all feature directories)
- [ ] activity-logs.resolved.yaml deleted
- [ ] No .gen.yaml or .resolved.yaml files remain anywhere in src/features/

```yaml
name: todos
label: Todo
labelPlural: Todos

kind: standard

fields:
  title:
    type: string
    required: true
    max: 200
  completed:
    type: boolean
    default: false
    filterable: true

timestamps: false

identity:
  type: auto-increment

behaviors:
  assignable: false
  orderable: true
  softDelete: false
  auditTrail: false
  immutable: false

access:
  scope: owner
  permissions:
    create: authenticated
    read: owner
    update: owner
    delete: owner

views:
  defaultView: list
  enabledViews: [list]
  filteredViews:
    - name: active
      label: Active
      filter: { completed: false }
      navEntry: false
    - name: completed
      label: Completed
      filter: { completed: true }
      navEntry: false

operations:
  create: true
  read: true
  update: true
  delete: true

indexes:
  - name: by_userId
    fields: [userId]
  - name: by_completed
    fields: [completed]
```

Note: No detailView, derivedData, hooks, or relationships needed — todos is intentionally the simplest possible feature spec. `userId` is auto-set by the generator's owner scope pattern.
