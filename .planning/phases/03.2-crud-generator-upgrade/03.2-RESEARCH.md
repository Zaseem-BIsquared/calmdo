# Phase 03.2: CRUD Generator Upgrade - Research

**Researched:** 2026-03-26
**Domain:** Code generation (Plop.js + Handlebars), YAML-driven configuration, Convex backend patterns, React component generation
**Confidence:** HIGH

## Summary

This phase transforms the existing Plop.js generators from TODO-stub scaffolders into a comprehensive, Frappe-level CRUD generation system. The generator takes a YAML feature definition and produces fully working code: Zod schema, Convex backend (mutations + queries + tests), frontend components (page, list, item, form, status badge), route file, i18n locale files, and auto-wiring of shared files (schema.ts, nav.ts, errors.ts, i18n.ts).

The core architecture is: **YAML definition -> resolver (merge defaults + feature overrides) -> Plop custom actions (template rendering + file modification) -> generated code**. Plop.js 4.0.5 (current, latest) supports custom action types, modify/append actions, and Handlebars helpers -- all needed for this. The YAML layer adds `yaml` package (2.8.3) for parsing and `deepmerge` (already a transitive dependency) for default merging.

The biggest risk area is **smart merge on re-run** (D-07). Phoenix and Rails scaffolds are one-shot -- they don't support re-running. The recommended approach is a **marked regions** strategy: generated code wrapped in `// @generated-start` / `// @generated-end` comments, with `// @custom-start` / `// @custom-end` blocks preserved during regeneration. This is simpler and more debuggable than AST-based merge.

**Primary recommendation:** Build the generator as a pipeline: `loadYAML -> resolveDefaults -> validateConfig -> generateActions`. Use Plop's custom action types for YAML resolution, file wiring (append to shared files), and marked-region-aware file writing. Keep templates as close to the tasks gold standard as possible -- the tasks feature IS the template reference.

## Architecture Patterns

### Generator Pipeline Architecture
```
Entry Points:
  Interactive wizard (plop prompts) ─┐
  Direct YAML file ──────────────────┤
                                     ▼
Pipeline:
  1. loadYAML() ── read {name}.gen.yaml (or create from prompts)
  2. resolveDefaults() ── deepmerge templates/defaults.yaml + feature YAML
  3. writeResolved() ── write {name}.resolved.yaml (committed, transparent)
  4. validateConfig() ── validate resolved YAML against expected schema
  5. generateActions() ── convert config into Plop action list
  6. executeActions() ── Plop runs add/modify/append/custom actions
```

### Pattern 3: Smart Merge via Marked Regions
**What:** Preserve customizations when re-running generator
**When to use:** All generated files
**Recommendation (Claude's discretion):** Use marked comment regions

```typescript
// @generated-start -- DO NOT EDIT between generated markers
import { mutation } from "@cvx/_generated/server";
import { auth } from "@cvx/auth";
// ... generated imports
// @generated-end

// @custom-start imports
// Add custom imports here -- preserved on re-run
// @custom-end imports

// @generated-start create
export const create = zMutation({
  args: createProjectInput,
  handler: async (ctx, args) => {
    const userId = await auth.getUserId(ctx);
    if (!userId) return;
    // @custom-start create-pre
    // Add pre-create logic here
    // @custom-end create-pre
    await ctx.db.insert("projects", { ...args, userId, position: Date.now() });
    // @custom-start create-post
    // Add post-create logic here
    // @custom-end create-post
  },
});
// @generated-end create
```

**Implementation approach:**
1. **On first run:** Generate file with `@generated-start`/`@generated-end` blocks containing template output, and empty `@custom-start`/`@custom-end` blocks at strategic hook points.
2. **On re-run:** Parse existing file, extract content from `@custom-start`/`@custom-end` blocks, regenerate the `@generated` blocks from new YAML config, re-inject the custom blocks.
3. **Plop custom action:** `setActionType("smartAdd", ...)` that reads existing file (if present), extracts custom regions, renders new template, injects preserved custom content.

**Advantages over AST-based merge:**
- No dependency on ts-morph or TypeScript compiler API
- Works for any file type (TS, TSX, JSON, YAML)
- Easy to debug -- just look at the comments
- Familiar pattern from C#/CodeSmith generators

**Limitations:**
- Users must not move custom blocks outside their markers
- Renaming a marker key loses that custom block
- Not as precise as AST manipulation

### Pattern 5: Handlebars Helpers for Template Logic
**What:** Register custom helpers for field-type-specific rendering
**When to use:** Templates need conditional logic based on field types

```javascript
// In plopfile.js
plop.setHelper("zodType", (field) => {
  switch (field.type) {
    case "string": return `z.string().max(${field.max || 200})`;
    case "text": return `z.string().max(${field.max || 5000})`;
    case "boolean": return `z.boolean().default(${field.default || false})`;
    case "number": return `z.number()`;
    case "enum": return `z.enum([${field.values.map(v => `"${v}"`).join(", ")}])`;
  }
});

plop.setHelper("convexType", (field) => {
  switch (field.type) {
    case "string": case "text": return "v.string()";
    case "boolean": return "v.boolean()";
    case "number": return "v.number()";
    case "enum": return `zodToConvex(${field.zodRef})`;
  }
});

plop.setHelper("ifEq", function(a, b, options) {
  return a === b ? options.fn(this) : options.inverse(this);
});
```

### Anti-Patterns to Avoid
- **Over-engineering the YAML schema up front:** Start with what tasks needs, expand from there. The YAML schema IS the tasks feature retroactively described.
- **Generating code that doesn't match the gold standard:** Every generated file must look like it was hand-written following the tasks pattern. If the template produces code that differs from how tasks does it, the template is wrong.
- **Trying to handle all behaviors in one template:** Use Handlebars partials/blocks. Each behavior (softDelete, orderable, assignable) should be a separate partial that gets conditionally included.
- **Making the resolver too smart:** The resolved YAML should be a simple deep merge. Don't add computed fields during resolution -- do that in the template helpers.
- **Generating files that need immediate manual editing to compile:** The generated code MUST pass typecheck and tests without any manual intervention. If it can't, the generator has a bug.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| YAML parsing | Custom parser | `yaml` npm package | Edge cases with multiline strings, anchors, comments |
| Deep object merging | Manual recursive merge | `deepmerge` | Array handling, prototype pollution protection |
| Template engine | String interpolation | Handlebars (via Plop) | Partials, helpers, escaping, conditionals |
| Form validation | Custom validators | Zod (generated schemas) | Composable, TypeScript inference, zodToConvex integration |
| Drag-and-drop | Custom implementation | @dnd-kit (already installed) | Touch support, keyboard accessibility, animation |
| CLI prompts | Custom readline | Inquirer (via Plop) | Validation, types (list, checkbox, confirm), autocomplete |
| File path manipulation | String concatenation | Plop's built-in path helpers | Cross-platform, Handlebars integration |

**Key insight:** The plopfile.js becomes significantly more complex -- it will need to import YAML, deepmerge, and implement custom action types. But Plop supports this natively. The plopfile should be split into modules: `generators/` directory with separate files for each generator, `generators/utils/` for shared helpers (YAML resolver, marked region parser, wiring helpers).

## Common Pitfalls

### Pitfall 1: Convex api.* Path Changes
**What goes wrong:** Moving or renaming files under `convex/` changes the `api.*` import paths that frontend code uses.
**Why it happens:** Convex generates `convex/_generated/api.d.ts` based on the file structure under `convex/`.
**How to avoid:** Generated code must use the exact pattern: `api.{featureName}.mutations.create`, `api.{featureName}.queries.list`. The feature name in the YAML must match the directory name under `convex/`.
**Warning signs:** TypeScript errors like "Property 'xxx' does not exist on type..." in frontend files.

### Pitfall 2: 100% Coverage Requirement Breaks Generation
**What goes wrong:** Generated code that isn't fully covered by generated tests causes the pre-commit hook to fail.
**Why it happens:** The vitest coverage config includes `src/features/**/*.{ts,tsx}` and `convex/**/*.ts`. Any generated file automatically falls under coverage tracking.
**How to avoid:** Generated tests must cover every branch in generated code. Use `/* v8 ignore start */` / `/* v8 ignore stop */` for genuinely untestable code (Radix portal interactions, dnd-kit drag events), with a comment explaining why.
**Warning signs:** Coverage report shows <100% on generated files.

### Pitfall 3: Handlebars Template Escaping
**What goes wrong:** Handlebars escapes HTML entities by default (`&`, `<`, `>`), which corrupts TypeScript code.
**Why it happens:** Handlebars was designed for HTML output.
**How to avoid:** Use triple-stache `{{{expression}}}` for code output, or register a "raw" helper. Plop's built-in helpers (pascalCase, camelCase, etc.) already work correctly.
**Warning signs:** Generated code contains `&amp;`, `&lt;`, or `&gt;`.

### Pitfall 4: Schema.ts Import Order Matters
**What goes wrong:** The auto-wiring action appends to `convex/schema.ts`, but the import for the Zod schema must come before the `defineSchema()` call.
**Why it happens:** Schema.ts has imports at the top and the schema definition below. Appending naively puts things at the wrong location.
**How to avoid:** Use Plop's `modify` action with a `pattern` to insert imports in the import section and table definitions inside the `defineSchema({})` call. Use separate patterns for each insertion point.
**Warning signs:** TypeScript errors about undefined variables in schema.ts.

### Pitfall 5: i18n Namespace Registration
**What goes wrong:** Generated components use `useTranslation("featureName")` but the namespace isn't registered in `src/i18n.ts`.
**Why it happens:** i18n.ts has a hardcoded `ns` array that must include the feature namespace.
**How to avoid:** Auto-wiring step must append the feature name to the `ns` array in `src/i18n.ts`. Use Plop's `modify` action with pattern matching.
**Warning signs:** Missing translations in the UI, console warnings about missing namespaces.

### Pitfall 6: Vitest Coverage Exclude Patterns
**What goes wrong:** Generated barrel files (`index.ts`) and route files are excluded from coverage by glob patterns in vitest.config.ts, but other generated files might not be.
**Why it happens:** The coverage config has explicit exclude patterns. New file patterns (like `.resolved.yaml`, new component types) might not be covered.
**How to avoid:** Check that generated files either (a) have tests covering them or (b) fall under existing exclude globs. The `.gen.yaml` and `.resolved.yaml` files are not TypeScript and won't be picked up.
**Warning signs:** Coverage report includes unexpected files with 0% coverage.

### Pitfall 7: TanStack Router File Naming Convention
**What goes wrong:** Generated route files don't follow the flat-file naming convention and TanStack Router's code generator doesn't pick them up.
**Why it happens:** TanStack Router uses a specific naming convention: `_layout.{segment}.tsx` for nested routes within a layout.
**How to avoid:** Route files must be named exactly `_layout.{featureName}.tsx` under `src/routes/_app/_auth/dashboard/`. The route path string must match: `"/_app/_auth/dashboard/_layout/{featureName}"`.
**Warning signs:** Route not appearing in the app, TanStack Router codegen not recognizing the file.

### Pitfall 8: Edge Runtime Test Environment
**What goes wrong:** Backend tests run in edge-runtime environment but use patterns that only work in jsdom.
**Why it happens:** Vitest config uses `environmentMatchGlobs: [["convex/**", "edge-runtime"]]`. Backend and frontend tests have different environments.
**How to avoid:** Generated backend tests must import from `../test.setup` (relative to convex/), use `test` fixture (not vitest's default), and not import DOM testing utilities. Generated frontend tests must import from `@cvx/test.setup` and use `renderWithRouter`.
**Warning signs:** "ReferenceError: document is not defined" in backend tests, or edge-runtime errors in frontend tests.

## Open Questions

1. **Plopfile module organization**
   - What we know: plopfile.js is currently 168 lines with 4 generators. It will grow significantly.
   - What's unclear: Best way to split. Plop supports `plop.load()` for loading from other files.
   - Recommendation: Create `generators/` directory. Each generator in its own file. Shared utils in `generators/utils/`. Use `plop.load()` or dynamic imports (ESM supported since type: "module").

2. **Marked region parser reliability**
   - What we know: Comment-based markers work well for simple cases.
   - What's unclear: How robust is regex-based extraction for complex nested code?
   - Recommendation: Keep markers at the top level only (function-level, not nested). Use unique marker names that include the operation: `@custom-start create-pre`, `@custom-start create-post`. Test the parser thoroughly.

3. **Interactive wizard field prompting UX**
   - What we know: Rails-style `name:string` shorthand is fast for experts.
   - What's unclear: How to implement the add/edit/remove loop for fields in Inquirer.
   - Recommendation: Use Inquirer's `editor` type for the field list (user types all fields at once, Rails-style), then show a confirmation table. For add/remove, use a recursive prompt loop with `list` type offering "Add field", "Edit field", "Remove field", "Done".

4. **GEN-05 not formally defined in REQUIREMENTS.md**
   - What we know: GEN-01 through GEN-04 are v1.0 requirements (satisfied). GEN-05 is referenced in the roadmap for this phase but not defined.
   - What's unclear: The exact acceptance criteria for GEN-05.
   - Recommendation: Treat GEN-05 as "Generator output is fully working CRUD that passes all tests at 100% coverage" per the phase success criteria. The planner should define this explicitly.
