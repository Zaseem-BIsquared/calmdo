<decisions>
## Implementation Decisions

### Generated/Custom Split Strategy
- **D-01:** Migrate ALL existing CalmDo features to the generated/custom split. Generated code goes to `src/generated/{name}/` and `convex/generated/{name}/`, custom code stays in `src/features/{name}/`. This proves the architecture works against real complexity.
- **D-02:** All-at-once migration (not incremental). One big pass, consistent codebase. Same approach that worked for v1.0 restructure.
- **D-03:** Existing 300+ tests must pass after migration. Test paths and imports updated in lockstep.

### Cross-Entity Panel Templates
- **D-04:** All 5 panel types delivered: related-record, table, checklist, timeline, cards. Full scope.
- **D-05:** Panels support inline editing for simple fields (title, status, assignee). Complex fields (description) open modal/detail page. Resolver already has `hasInlineEditing` flag — templates must respect it.
- **D-06:** CalmDo mapping: subtasks=checklist, work-logs=timeline, activity-logs=timeline, project-tasks=table. Cards panel used for project overview.

### End-to-End Pipeline Testing
- **D-07:** Both unit tests AND integration tests. Unit tests mock filesystem/git for edge cases. Integration tests run the full pipeline against real feature YAMLs.
- **D-08:** Two test fixtures: todos (simple, fast CI) and CalmDo (complex, thorough validation). Todos runs on every commit, CalmDo runs as extended validation.
- **D-09:** "Tested" means: `feather generate` produces valid code from YAML, `feather update` hard-overwrites generated/ without touching custom/, uncommitted change detection works, --dry-run shows what would change, --force bypasses uncommitted check.

### CLI Completion
- **D-10:** Project-local only (`npx feather` or `npm run feather`). No global install. Avoids version conflicts.
- **D-11:** `feather init` not needed separately — `npm create feather` (Phase 999.2) handles new projects. For now, init exists as internal scaffolding only.
- **D-12:** package.json `bin` entry must point to `bin/feather.ts` so `npx feather` works from any project that has the dependency.

### Claude's Discretion
- Migration execution order (schema first, then backend, then frontend — or feature-by-feature)
- Panel template styling (reuse existing Radix/Tailwind patterns from CalmDo components)
- Test fixture structure (inline YAML strings vs fixture files)
- Whether `feather update` integration test creates a temp git repo or uses a snapshot approach

</decisions>

## Existing Code Insights

### Reusable Assets
- **`templates/pipeline/`**: Full pipeline exists (generate, wire, scaffold). Needs retargeting from `src/features/` to `src/generated/` + `convex/generated/`.
- **`templates/cross-entity/resolver.ts`**: Resolver reads YAML `detailView.relatedRecords` config. Already handles `hasInlineEditing` flag. Only panel rendering templates are missing.
- **`bin/commands/update.ts`**: Update command exists with `findFeatureYamls()`, `hasUncommittedChanges()`, dry-run, force. Already looks in both `src/features/` and `src/generated/` for YAMLs.
- **26 Handlebars templates**: All single-entity templates exist. Cross-entity templates are the gap.
- **8 feather.yaml files**: Real YAML specs for all CalmDo + example entities.

### Established Patterns
- **Feature-folder architecture**: `src/features/{name}/` + `convex/{name}/` — clear split boundary.
- **`outputMode` flag**: `bin/commands/generate.ts` already supports `legacy` vs `generated` output modes. The split infrastructure is partially wired.
- **Zod validation**: YAML validated by Zod schema. Same validators used at generation time.

### Integration Points
- **`convex/schema.ts`**: Must import from `convex/generated/{name}/` after migration (currently from `convex/{name}/`).
- **Route files**: `src/routes/` imports from features — needs updating to import from generated or custom.
- **Shared wiring files**: `src/shared/nav.ts`, `src/shared/errors.ts`, `src/i18n.ts` — AST wiring already handles these.
- **Test imports**: 300+ tests reference `src/features/` and `convex/` paths — all need updating.

</code_context>

<specifics>
## Specific Ideas

- The all-at-once migration mirrors the successful v1.0 restructure approach (65 commits, 181 files). User has confidence in big-bang migrations.
- CalmDo as the primary integration test fixture validates the full complexity spectrum (5 entities, relationships, behaviors, cross-entity views).
- Todos as the fast CI fixture catches regressions without the overhead of full CalmDo generation.

</specifics>
