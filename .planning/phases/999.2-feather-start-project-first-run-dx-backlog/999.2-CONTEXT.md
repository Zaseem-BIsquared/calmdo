# Phase 999.2: `feather start project` — First-Run DX — Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

A developer runs `npm create feather my-project` and gets a working Feather project with branding, configured auth, selected example apps, and a ready-to-use generator. The command uses a hybrid approach: clone a minimal base branch + scaffold selected features.

</domain>

<decisions>
## Implementation Decisions

### Project Creation Flow
- **D-01:** Primary command: `npm create feather my-project` — standard npm convention using `create-feather` package.
- **D-02:** 3-question interactive wizard:
  1. **Project name** (required, no default)
  2. **Auth providers** — multi-select from: Password (default, zero-config), OTP (auto-enabled in dev via dev mailbox), GitHub OAuth (needs OAuth app credentials). Default: Password only.
  3. **Example apps** — multi-select from existing apps. Default: install all. Also available later via `feather add`.
- **D-03:** App display name derived from project name (my-project → My Project). No separate description question.
- **D-04:** `feather add` command allows installing example apps and features after creation — users who skip in the wizard can add later.

### Example App Strategy
- **D-05:** Default: install ALL existing example apps for browsing and learning.
- **D-06:** Example apps form a complexity ladder: todos (simple) → tickets (medium, status transitions) → contacts (medium, multi-field) → CalmDo (complex, relationships).
- **D-07:** Each example app is independently installable/removable via `feather add` / `feather remove`.

### Auth Provisioning
- **D-08:** Password provider works zero-config in development (Convex handles it).
- **D-09:** OTP auto-enabled in dev mode — dev mailbox intercepts emails, no Resend key needed for local development. Resend key prompted only at deploy time.
- **D-10:** GitHub OAuth prompts for client ID + secret only when user selects it in the wizard. If skipped, provider is disabled with instructions to enable later.
- **D-11:** Auth provider selection modifies `convex/auth.ts` to include/exclude providers based on wizard choices.

### Template vs Scaffold (Hybrid Approach)
- **D-12:** Hybrid: clone a minimal `base` branch (auth, shared infra, UI, test setup — no features), then scaffold selected features/examples via `feather add`.
- **D-13:** `base` branch auto-generated from `main` by a CI job that strips feature code. Always in sync with main. Feature-folder architecture provides clear boundaries for stripping.
- **D-14:** After clone + scaffold, squash-reset for clean git history (one initial commit).
- **D-15:** `create.sh` (curl approach) kept as fallback for environments without npm, but `npm create feather` is the primary path.

### Claude's Discretion
- CI implementation for base branch auto-generation (GitHub Actions workflow details)
- `feather add` command internals (uses existing generator pipeline or needs new logic)
- Package publishing strategy for `create-feather` npm package
- Post-creation verification step (run `npx convex dev --once` to confirm setup)

</decisions>

<specifics>
## Specific Ideas

- The 3-question wizard maps to Tableau Standard: only ask when genuinely variable (name, auth, examples)
- Dev mailbox makes OTP zero-friction in development — brilliant DX already built
- `feather add` as a post-creation escape hatch means the wizard can default to "install all" without anxiety

</specifics>

<deferred>
## Deferred Ideas

- **Header-detail showcase app** — User wants an example app demonstrating parent-child relationships (invoice/line-items, leave management, purchase orders). Current examples don't showcase this pattern. Should be a new phase after the generator supports `children:` relationship type fully.
- **CalmDo as installable example** — CalmDo (tasks+projects+subtasks+work-logs+activity-logs) should be installable as a single "project management" example. Needs `feather add calmdo` that installs all 5 entities + wiring.
- **Example app onboarding workflow** — Ideal learning path: See UI → play with it → read config → understand generation → start own project. Phase 999.2 should include a generated README or welcome page that guides this journey.

</deferred>
