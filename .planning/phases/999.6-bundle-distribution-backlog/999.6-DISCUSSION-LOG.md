## Bundle Manifest Format

| Option | Description | Selected |
|--------|-------------|----------|
| Feature list + metadata | Bundle = named list of features + description + complexity. Cross-entity wiring stays in feather.yaml. Simple, no duplication. | ✓ |
| Feature list + install order + shared config | Adds explicit installOrder and sharedConfig (nav group, search indexes). More expressive but heavier. | |

**User's choice:** Feature list + metadata
**Notes:** Cross-entity wiring already declared in individual feather.yaml `detailView.relatedRecords` — no need to duplicate in bundle manifest.

### Bundle Directory Location

| Option | Description | Selected |
|--------|-------------|----------|
| templates/bundles/ (separate) | Clean namespace, no collision risk with features | ✓ |
| templates/examples/ with type field | Same directory, manifest gains type field | |

**User's choice:** Separate `templates/bundles/` directory

---

## Terminology

User raised that "bundles" and "examples" is confusing. Discussion concluded:
- **Feature** = atomic installable unit (whether pre-built or custom)
- **Bundle** = named collection of features
- **Example** = documentation label only, not a CLI concept
- `templates/examples/` to be renamed to `templates/features/`

## Distribution Channel

User requested deeper research before deciding. Research conducted across 5 distribution models:

| Model | Source | Key Insight |
|-------|--------|-------------|
| shadcn v3 registry | Static JSON at URL, CLI fetches + copies | Closest fit — no npm dependency, files are copied |
| Frappe bench | Git repo per app, `bench get-app <url>` | Too heavy — features aren't standalone repos |
| WordPress plugins | Centralized marketplace with ratings | Way too heavy for current stage |
| Laravel/Packagist | npm/Composer packages | Overkill for v0 |
| Statamic starter kits | Git repo + manifest YAML | Close to `feather start project`, not `feather add` |

Three models proposed:

| Option | Description | Selected |
|--------|-------------|----------|
| Model A: GitHub Raw as Registry | Repo IS the registry, fetch from raw.githubusercontent.com | |
| Model B: Template Sync from Upstream | Git remote + selective fetch of templates/ | |
| Model C: Hybrid (local + GitHub fallback) | Check local first, fallback to registry URL, cache locally | ✓ |

**User's choice:** Model C — Hybrid
**Notes:** User's scenario is team distributing features to clients from a known GitHub repo. Key requirement: clients need to pull new features AND get bug fixes/updates after initial project creation. User confirmed GitHub-based mechanisms are workable. Bug fixes flow naturally through `feather update` which syncs from registry then regenerates.

## Dependency Resolution

| Option | Description | Selected |
|--------|-------------|----------|
| Topological sort from feather.yaml | Read belongs_to, build DAG, derive install order automatically | ✓ |
| Explicit installOrder in bundle.json | Bundle author declares order manually | |

**User's choice:** Topological sort
**Notes:** User asked for explanation of topological sort. Concept explained with CalmDo dependency graph example. Key advantage: can't get out of sync with actual feather.yaml declarations.

### Already-Installed Features

| Option | Description | Selected |

**User's choice:** Update existing + install missing
