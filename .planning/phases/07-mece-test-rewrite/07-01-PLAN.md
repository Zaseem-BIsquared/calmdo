# Plan 07-01: Backend Test Rewrite — MECE Compliance

## Objective

Rewrite all 16 backend test files (`convex/**/*.test.ts`) to follow the feather-testing-convex testing philosophy: MECE state decomposition, philosophy-compliant naming, proper seed patterns, and multiple assertions per state.

## Why This Matters

Backend tests are the foundation — they verify auth guards, validation, side effects (activity logs), cascade deletes, and edge cases that frontend integration tests can never reach. These tests are NOT redundant with frontend tests (see research Issue B1). The rewrite standardizes naming, adds test matrix comments, and ensures MECE compliance without losing any coverage.

## must_haves
- [ ] All 16 backend test files pass `npm test` with 100% coverage after rewrite
- [ ] Each test file has a MECE test matrix comment at the top
- [ ] Test names follow `"verb what-user-sees [condition]"` convention
- [ ] No `toBeDefined()` or `toBeTruthy()` assertions without content verification
- [ ] No `expect(true).toBe(true)` placeholder tests
- [ ] No snapshot tests
- [ ] Raw `ctx.db.insert()` used only when testing the mutation itself (not for query test setup)

<action>
Rewrite `convex/tasks/mutations.test.ts` and `convex/tasks/queries.test.ts`:

1. Add MECE test matrix comment at top of each file listing all states/behaviors covered.

2. For `mutations.test.ts`, organize by mutation function. Each mutation gets a describe block with states:
   - `create`: creates with defaults, creates with all fields, does nothing when unauthenticated, creates activity log
   - `update`: updates fields, does nothing when unauthenticated, logs edited/status_changed/assigned/unassigned activity
   - `remove`: deletes task, cascades subtasks + work logs + activity logs, does nothing when unauthenticated, logs deleted activity
   - `updateStatus`: transitions status, does nothing when unauthenticated, logs status_changed
   - `assignTask`: assigns to user, auto-flips visibility to shared, does nothing when unauthenticated, logs assigned
   - `unassignTask`: removes assignee, auto-flips visibility to shared, logs unassigned
   - `reorder`: updates position, does nothing when unauthenticated
   - `createInProject`: creates task linked to project, inherits project visibility

3. Rename all tests to philosophy convention:
   - BAD: `"creates a task with default values"` → GOOD: `"creates task with default status, visibility, and priority"`
   - BAD: `"does nothing when unauthenticated"` → GOOD: `"rejects unauthenticated create"` or `"silently ignores unauthenticated call"`

4. For `queries.test.ts`, organize by query function:
   - `myTasks`: returns user's assigned tasks, sorts by position, returns empty for unauthenticated
   - `teamPool`: returns unassigned shared/project tasks, excludes private tasks, returns empty for unauthenticated

5. Replace raw `ctx.db.insert("tasks", {...})` in query tests with `client.mutation(api.tasks.mutations.create, {...})` where possible. Keep raw inserts only when you need to set up data that bypasses mutation logic (e.g., tasks owned by other users).

6. Consolidate duplicate assertions: if two tests verify the same state from different angles, merge them into one test with multiple assertions.

Run `npm test` after the rewrite to verify 100% coverage maintained.
</action>

- `npm test` exits 0 with 100% coverage
- No `toBeDefined()` used as sole assertion
