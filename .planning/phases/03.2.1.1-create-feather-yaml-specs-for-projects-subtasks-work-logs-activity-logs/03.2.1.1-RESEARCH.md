## Pattern: Complete feather.yaml Sections

From `tasks/feather.yaml` (gold standard), every complete spec should have:

```yaml
# 1. Identity
name: <camelCase>
label: <Singular>
labelPlural: <Plural>

# 2. Fields (all field definitions)
fields:
  <field>:
    type: <type>
    required: <bool>
    max: <number>
    # ... type-specific options

# 3. Timestamps
timestamps: false  # or true/both

# 4. Identity (auto-increment, uuid, etc.)
identity:
  type: auto-increment

# 5. Behaviors
behaviors:
  assignable: <bool>
  orderable: <bool>
  softDelete: <bool>
  auditTrail: <bool>

# 6. Access
access:
  scope: <owner|authenticated|custom>
  permissions:
    create: <authenticated|owner|custom>
    read: <owner|authenticated|custom>
    update: <owner>
    delete: <owner>

# 7. Views
views:
  defaultView: <list|card|table>
  enabledViews: [list, card, table]
  filteredViews: [...]  # optional

# 8. Relationships (optional)
relationships:
  <name>:
    type: <belongs_to|has_many|children|embedded>
    target: <entity>
    # ...

# 9. Detail View (optional — TODO: generator pending)
detailView:
  layout: <panels|tabs|sections>
  relatedRecords: [...]

# 10. Derived Data (optional — TODO: generator pending)
derivedData:
  <name>:
    type: <count|sum|avg>
    source: <entity>

# 11. Hooks (optional — TODO: generator pending)
hooks:
  afterSave: <path>
  onStatusChange: <path>

# 12. Indexes
indexes:
  - name: <name>
    fields: [<fields>]
```

## Schema Ground Truth (from convex/schema.ts)

Actual fields in production — specs must match:

| Entity | Fields in schema.ts |
|--------|-------------------|
| tasks | title, description?, priority(bool), status, visibility, creatorId, assigneeId?, projectId?, position |
| projects | name, status, creatorId |
| subtasks | title, status, taskId, position, promotedToTaskId?, creatorId |
| workLogs | body, timeMinutes?, taskId, creatorId |
| activityLogs | entityType, entityId(string), action, actor(id users), metadata? |
| todos | title, completed?(bool), userId, position |
| tickets | title, description?, status, priority(enum), userId, assigneeId?, position |
| contacts | name, email?, company?, status, phone?, userId |

**Important:** `activityLogs.actor` is already `v.id("users")` in schema.ts, so feather.yaml should reflect `belongs_to` (D-13 is aligned with reality).
